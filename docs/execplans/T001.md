## Decomposition — Slice 1

### One-sentence objective

Extend the existing `inspect-db` CLI with three read-only views
(`job_claims`, `job_selection_signals`, `job_selection_assessments`) and an
optional `--job-id` filter, so the already-persisted claim and selection layer
becomes auditable from the operator console without any schema, extractor, or
pipeline change.

### Why this slice is first

- `PRODUCT_VISION.md` lists `job claims` as near-term foundation item 1, and
  the claim layer itself already exists in both the pydantic models
  (`jobpipe/decision/models.py`) and the primary DB (`job_claims`,
  `job_selection_signals`, `job_selection_assessments` in
  `jobpipe/core/primary_db.py` lines ~386–475).
- The open gap inside `specs/job-claims-model.md` §"First implementation slice"
  item 7 is *"inspection output so the extracted claims can be audited"*.
  Closing that gap first is what turns the already-persisted claim layer from
  an invisible internal artifact into a candidate-first, hiring-aware
  observability surface — which is exactly the `Now` priority of dashboard /
  operator trustworthiness on `ROADMAP.md`.
- It is the smallest candidate-first, local-first step that advances the job
  catalog foundation without touching schema, pipeline semantics, auth,
  billing, deploy, secrets, or the OSS/Workbench boundary.
- It adds no new runtime dependencies: the `inspect-db` entry point already
  uses stdlib `argparse`, `sqlite3`, and `json`, and already reads the primary
  DB via `connect_primary_db`.

### Files expected to be touched / created

- edit `jobpipe/cli/inspect_primary_db.py` — add three view helpers, extend the
  `--show` choice list, add an optional `--job-id` argument used by the three
  new views only, keep all existing views unchanged.
- new `tests/test_inspect_primary_db_claims.py` — unit tests for the three new
  views (happy path, empty-but-valid-job path, unknown-job path, `--json`
  shape).
- edit `docs/cli.md` — extend the existing `inspect-db` example block with one
  line showing the three new `--show` values plus `--job-id`.

### Files explicitly out of scope

- `jobpipe/core/primary_db.py` — no schema change, no new index, no new write
  path. Escalate if this slice appears to require touching schema.
- `jobpipe/decision/derive.py` — extractor output is unchanged in this slice.
- `jobpipe/decision/persistence.py` — write path is unchanged.
- `jobpipe/cli/main.py` — `inspect-db` is already registered; no subcommand
  change is needed.
- `jobpipe/stages/*`, `jobpipe/projections/*`, `jobpipe/runtime/*` — pipeline,
  projection, and runtime paths untouched.
- `AUDIT.md`, `AGENT_STATUS.md` — historical, not canonical.
- `specs/job-claims-model.md` — flagged mismatch below; a follow-up
  docs-hygiene slice should reconcile it, not this one.

### Acceptance criteria

- `python -m jobpipe.cli.main inspect-db --show job_claims --job-id <id>`
  prints only rows scoped to that `job_id`.
- `python -m jobpipe.cli.main inspect-db --show job_claims` (no `--job-id`)
  prints up to `--limit` rows (default 20), ordered by
  `importance_score DESC, updated_at DESC`.
- `--show job_selection_signals` and `--show job_selection_assessments`
  behave analogously and honor `--job-id` and `--limit`.
- When `--job-id` is supplied but the job is not present in the `jobs` table,
  the command exits non-zero with a clear stderr message. Do not silently
  return an empty row list for an unknown job.
- When the job exists in `jobs` but has zero rows in the requested view,
  the command exits 0 with a single informational line, for example
  `[job_claims] no rows for job_id <id>`.
- `--json` output for each new view is a parseable JSON list of row dicts
  shaped like the existing `--show jobs` / `--show evaluations` JSON blocks.
- No existing `--show` value changes in behavior or output.
- No new runtime dependency is introduced.
