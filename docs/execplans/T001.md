# T001 - Job Catalog Foundation

## Goal

Continue the JobPipe job catalog foundation work from the clean dual-client
orchestrator baseline.

## Why this exists

The dual-client structure is now committed and isolated in v2 worktrees. The
next useful work should return to JobPipe's product/runtime foundation instead
of continuing instruction cleanup.

## Scope

- Use Claude on `../Jobpipe-orchestrator-v2` to classify the next job catalog
  foundation step.
- Use `../Jobpipe-claude-v2` for optional deeper planning or review.
- Use Codex on `../Jobpipe-codex-v2` only after Claude produces an approved
  one-step worker prompt.
- Keep the task candidate-first, local-first, and aligned with the public OSS
  foundation direction.
- Preserve runtime traceability and documented validation evidence.

## Out of Scope

- More instruction-system restructuring.
- User-level Codex or Claude config changes.
- MCP activation.
- Historical source archival.
- Broad refactors without a specific approved slice.
- Runtime data deletion or full source-queue rebuilds.

## Validation Guidance

The exact tests depend on Claude's selected one-step objective. Default expected
validation for code changes is:

- the most relevant targeted pytest command
- `python compile_check.py`
- `jobpipe run --dry-run` or `python -m jobpipe.cli.main run --dry-run` when the
  touched behavior affects the main runtime path

Docs-only planning changes should validate cross-doc consistency instead of
pretending runtime tests are relevant.
