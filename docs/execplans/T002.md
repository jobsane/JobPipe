**Written:** 2026-04-21

## Why this exists

Candidate-facing end-goal (from founder): "actually use, decide on what jobs to
apply to, apply for them, and track them with documents and all stored in a
structured way so i don't get filedumps everywhere."

The authoring MVP is the last missing piece between "JobPipe has structured
decisions" and "candidate presses apply with the right document." Everything
upstream of this (intake, claims, evidence, narrative, decision tables) already
exists in enough form to consume.

## Position decision (locked)

**Option C (hybrid).** Deterministic contracts + validation stay JobPipe-native.
crewAI enters only inside the author/revise agent layer, behind a JobPipe-owned
adapter. Contracts must be **agent-runtime-swappable**.

Concretely that means:

- `AuthoringCaseContext`, `GeneratedApplicationPackage`, `DocumentValidationResult`
  are plain Python dataclass / pydantic shapes. They do not import anything
  from `crewai`. Ever.
- The author/revise layer is a single module with a typed interface. The first
  implementation may be a simple prompt loop; a later implementation may use
  crewAI; swapping is a one-file change.
- Evidence selection, claim targeting, narrative constraints, and validation
  rules all run on JobPipe-owned code, not inside the agent framework.

A slice that imports `crewai` into the contract layer is a routing violation
and should be rejected in review.

## Constraints the planner must encode in the slice brief

1. **Contract purity**: `AuthoringCaseContext` lives in a module under
   `jobpipe/model/` or `jobpipe/authoring/`. It must not import any agent
   framework. The planner must state this in acceptance criteria.
2. **Reuse over re-derive**: the context should be *built from* existing
   canonical state (candidate, job evaluation, evidence units, narrative
   brief, artifact plan) — not a parallel truth model. The audit step must
   map each field to its existing source of truth before definition.
3. **Small, reversible**: single file + tests. If the audit surfaces that
   `application_pack` code already owns parts of this shape, the planner
   should propose extending over replacing.
4. **Test style**: mirror `tests/test_inspect_primary_db_claims.py` — plain
   pytest, no heavyweight fixtures. Respect the Python 3.14 + anyio workaround
   already in `docs/current-state.json`:
   `PYTEST_DISABLE_PLUGIN_AUTOLOAD=1 -p no:debugging -p no:cacheprovider`.
5. **Acceptance criteria must include**: "no `crewai` import in the module or
   its tests," tested via an import assertion or a simple grep check in CI.

## Sprint 2 — Author/revise module + first end-to-end generation pass

**Sprint goal:** Close the loop from `AuthoringCaseContext` to a persisted
`GeneratedApplicationPackage` (cover-letter draft + CV projection + validation
result). No crewAI. Uses the existing `agents.Agent` (openai-agents SDK)
pattern already established in `jobpipe/stages/application_pack.py`.

**Date scoped:** 2026-04-22
**Routing decision:** Opus coordinator (T002.md §Sprint 2)

### crewAI position (locked for this sprint)

The codebase already uses `from agents import Agent` (openai-agents SDK).
crewAI is a separate dependency currently in parked issues #82–#89 — it requires
a security/dependency spike (#86) which stays backlog until the local MVP works
end-to-end. Sprint 2 uses the existing `agents.Agent` pattern.

The author adapter interface **must be typed and swappable** so a future crewAI
(or other) implementation is a one-file drop-in. This is the only architectural
constraint on the adapter design.

### Slices (ordered)

1. **Slice 6 — `--validate` flag on `build-authoring-context` CLI**
   Issue: #63-followup (deferred from Slice 5 brief)
   Scope: add `--validate` flag to the smoke CLI in
   `jobpipe/authoring/smoke_cli.py`. Calls `validate_authoring_context` on the
   built context and prints the result. Pure offline, no LLM. Small.

2. **Slice 7 — Author adapter protocol + cover-letter + CV-projection author**
   Issues: #66, #67, #71
   Scope: new `jobpipe/authoring/author.py`.
   - Define `AuthorAdapter` as a `typing.Protocol`:
     `def author(self, ctx: AuthoringCaseContext) -> GeneratedApplicationPackage`
   - Implement `SimpleAgentAuthor` using `agents.Agent` (same SDK + pattern as
     `application_pack_stage_factory`). Generates:
     - `cover_letter_draft: str` — full prose cover letter in Norwegian
     - `tailored_cv_projection: dict` — structured CV highlights projection
       (follows `ReactiveResumeTailoredCVProjection` shape from `model/schema.py`)
     - `evidence_refs: list[dict]` — evidence units referenced
     - `gap_notes: list[str]` — disclosed gaps / mitigations
   - Runs `validate_authoring_context` after generation; stores serialised result
     in `GeneratedApplicationPackage.validation`.
   - No imports from crewai, langchain, autogen.
   - No modification to `AuthoringCaseContext`, `GeneratedApplicationPackage`,
     or `DocumentValidationResult`.

3. **Slice 8 — Persistence + `jobpipe author-package` CLI hook**
   Issues: #64, #75, #77
   Scope: new `jobpipe/authoring/persist.py` + new CLI command.
   - `persist_generated_package(pkg, *, candidate_id, job_id, evaluation_id)`:
     saves JSON to `JOBPIPE_DATA_DIR/documents/<candidate_id>/<job_id>/`
     and inserts row via existing `insert_generated_document` in
     `jobpipe/core/primary_db.py`.
   - `jobpipe author-package --job <id>`: builds context → authors → validates
     → persists → prints summary (passed/failed, score, cover letter first 200
     chars).

### Sprint 2 exit criteria
